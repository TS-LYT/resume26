# TCP 三次握手、四次挥手

## 三次握手是什么

TCP 建立连接需要三次握手：

```text
1. 客户端 -> 服务端：SYN
   客户端告诉服务端：我想建立连接，我的初始序列号是 x。

2. 服务端 -> 客户端：SYN + ACK
   服务端告诉客户端：我收到你的 SYN 了，我也想建立连接，我的初始序列号是 y。

3. 客户端 -> 服务端：ACK
   客户端告诉服务端：我收到你的 SYN 了，连接建立。
```

## 为什么不是两次握手

两次握手只能确认“客户端能发、服务端能收，服务端能发”，但服务端无法确认“客户端能收”。第三次 ACK 是为了让服务端确认客户端也能接收数据。

面试口述：

```text
三次握手的核心是让双方都确认对方的发送和接收能力正常，并同步双方的初始序列号。
```

## 四次挥手是什么

TCP 断开连接通常需要四次挥手：

```text
1. 主动关闭方 -> 被动关闭方：FIN
   我这边没有数据要发了。

2. 被动关闭方 -> 主动关闭方：ACK
   我知道你不发了，但我可能还有数据没发完。

3. 被动关闭方 -> 主动关闭方：FIN
   我这边数据也发完了，可以关闭了。

4. 主动关闭方 -> 被动关闭方：ACK
   我收到你的 FIN 了。
```

## 为什么挥手通常是四次

因为 TCP 是全双工的。客户端不发数据了，不代表服务端也马上没有数据要发。所以关闭两个方向的数据流，通常需要分别确认。

## TIME_WAIT 是什么

主动关闭连接的一方会进入 `TIME_WAIT`。它的主要作用：

```text
1. 确保最后一个 ACK 能被对方收到。
2. 让旧连接里的延迟报文在网络中自然消失，避免影响后续同四元组连接。
```

## 常见面试追问

### 1. SYN flood 是什么？

攻击者发送大量 SYN，但不完成第三次握手，服务端半连接队列被占满，正常连接可能受影响。

### 2. close 和 shutdown 区别？

`close(fd)` 关闭文件描述符；如果引用计数为 0，会关闭连接。  
`shutdown(fd, SHUT_WR)` 可以只关闭写方向，仍然允许读，适合半关闭场景。

### 3. 服务端大量 TIME_WAIT 怎么办？

首先判断为什么服务端是主动关闭方。通常应该让客户端主动关闭。也可以检查连接复用、长连接、keepalive、端口耗尽等问题。

## 和 socket 代码的关系

应用层代码通常看不到 SYN/FIN 包细节：

```text
connect() 触发三次握手
accept() 返回表示握手已经完成
close() 触发挥手
```

```bash
sudo tcpdump -i lo tcp port 9000 -nn
```
